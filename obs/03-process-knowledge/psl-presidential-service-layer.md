# PSL — Presidential Service Layer 总裁直属服务体系

## 一、体系定位

PSL 是 Synapse 体系中**直接服务总裁个人**的服务层，与团队管理（Lysander CEO统筹）平行运作。

```
Synapse 体系全景：

  总裁 刘子杨
    │
    ├── PSL（直接服务总裁个人）
    │   ├── SPE 条线：帮总裁规划"要做什么"
    │   │   ├── /capture — 快速任务捕获
    │   │   ├── /plan-day — 每日规划
    │   │   ├── /time-block — 时间块安排
    │   │   └── /weekly-review — 每周回顾
    │   │
    │   └── PBS 条线：帮总裁看到"做了什么" + 展示"做了什么"
    │       ├── 3x/天定时简报（午报/晚报/夜报）
    │       ├── 成就组合自动累积
    │       ├── 分享/演示资料生成
    │       └── 风格学习与偏好模型
    │
    └── Lysander CEO（管理团队执行）
        ├── Graphify 智囊团
        ├── Butler / RD / OBS / Content Ops / ...
        └── 四级决策体系
```

## 二、设计理念

### 2.1 SPE + PBS 的互补关系

| 维度 | SPE（Synapse Personal Engine） | PBS（Presidential Briefing Service） |
|------|-------------------------------|--------------------------------------|
| 核心问题 | "今天要做什么？" | "今天做了什么？怎么展示？" |
| 数据流向 | 外部信息 → 规划 → 总裁 | 执行结果 → 汇总 → 总裁 |
| 时间维度 | 前瞻（规划未来） | 回顾（总结过去）|
| 输出物 | 日程/任务/时间块 | 简报/成就快照/演示资料 |
| 交互频率 | 按需（总裁发起） | 定时推送（系统发起） |

### 2.2 共享基础设施

两条线共享以下数据源和基础设施：
- `personal_tasks.yaml` — 个人任务数据
- `active_tasks.yaml` — 团队任务数据
- `briefing_style_guide.yaml` — 总裁风格偏好模型
- OBS 知识库 — 方法论和决策记录
- 情报日报管线 — 行业动态
- Google Calendar / Slack — 日程和沟通数据

## 三、PBS 条线设计

### 3.1 定时简报架构

```
07:00  晨报 ← 与SPE 6:15am Agent合并（不新增）
12:00  午报 ← executive_briefer 生成
18:00  晚报 ← executive_briefer 生成
22:00  夜报 ← executive_briefer 生成
```

### 3.2 简报三层递进阅读设计

| 层次 | 阅读时间 | 内容 | 设计依据 |
|------|---------|------|---------|
| 第一层 BLUF | 10秒 | 一句话核心结论 | 军事简报BLUF原则 |
| 第二层 要点 | 1分钟 | 3-5个金字塔结构要点 | McKinsey金字塔原理 |
| 第三层 详情 | 5分钟 | SCQA完整叙事+数据支撑 | Amazon 6-Page Memo |

### 3.3 各时段简报内容定义

**午报（12:00）— 进度追踪**
- BLUF：上午进展一句话总结
- 上午完成项（权重50%）
- 异常与调整（权重30%）
- 下午重点（权重20%）

**晚报（18:00）— 成就回顾**
- BLUF：今日最大成就一句话
- 今日叙事成就（权重40%，用SAIR框架）
- 趋势洞察（权重30%）
- 明日预告（权重30%）

**夜报（22:00）— 规划反思**
- BLUF：明天最该期待什么
- 明日关键任务（权重40%）
- 本周进度与目标关联（权重30%）
- 反思与调整建议（权重30%）

### 3.4 数据源映射

| 数据源 | 简报板块 | 采集方式 |
|--------|---------|---------|
| active_tasks.yaml | 进度追踪/完成项 | 直接读取 |
| personal_tasks.yaml | 个人任务进展 | 直接读取 |
| git log | 今日代码/配置变更 | 命令行采集 |
| 远程Agent执行日志 | 自动化产出 | git pull后读取 |
| 情报日报 | 行业洞察板块 | 读取daily-intelligence/ |
| Google Calendar | 明日议程 | MCP读取 |
| Slack | 团队动态/反馈 | MCP读取 |

## 四、风格学习机制

### 4.1 四通道反馈

| 通道 | 优先级 | 处理方式 | 生效速度 |
|------|--------|---------|---------|
| 口头指令 | 最高 | 直接写入explicit_rules | 立即 |
| 内联修正 | 高 | Diff提取修改模式 | 下次生效 |
| 即时评价 | 中 | 更新权重 | 累积后调整 |
| 行为推断 | 低 | 统计建模 | 周期性调整 |

### 4.2 收敛路径

```
cold_start (0-5次反馈)
  → 基本格式/长度偏好定型
  
calibrating (6-15次反馈)
  → Style Guide覆盖80%，大部分内容无需大改
  
converging (16-25次反馈)
  → 修改率持续下降
  
stable (25+次反馈)
  → 修改率<5%，进入维护模式
```

### 4.3 Style Guide 文件

配置文件：`agent-butler/config/briefing_style_guide.yaml`
- 内容偏好（板块权重、时段差异化）
- 格式偏好（长度、结构、可视化方式）
- 语言风格（正式度、用词偏好、禁用词）
- 简报模板结构（三层递进参数）
- 反馈规则库（总裁明确指令）
- 收敛追踪（当前阶段、反馈次数、满意度曲线）

## 五、成就组合系统

### 5.1 自动累积

每个M/L级任务完成后自动提取为成就条目，包含：
- 成就标题 + SAIR叙事
- 量化影响指标
- 关联的OKR/战略目标
- 时间戳和负责团队

### 5.2 汇聚节奏

```
每日简报（3x/天） → 日成就
每周快照（1x/周） → 周度成就TOP5
每月报告（1x/月） → 月度战略进展
按需演示（事件驱动）→ 分享/演示资料
```

### 5.3 叙事转化规则

| 任务类型 | 叙事模板 | 价值维度 |
|----------|---------|---------|
| 功能开发 | "推出了[功能]，使[能力]成为可能" | 能力扩展 |
| Bug修复 | "解决了[问题]，消除了[风险]" | 风险降低 |
| 架构优化 | "重构了[组件]，提升了[质量属性]" | 技术债务降低 |
| 流程改进 | "优化了[流程]，使[效率指标]提升" | 效率提升 |
| 知识沉淀 | "建立了[知识体系]，为[未来能力]奠定基础" | 知识资产 |

## 六、Agent 编制

|------|--------------|---------|
| 高管简报师 | executive_briefer | 简报生成、数据融合、成就转化、定时推送 |
| 风格校准师 | style_calibrator | 风格建模、反馈处理、Style Guide维护、一致性审计 |

- integration_qa → 简报质量门禁
