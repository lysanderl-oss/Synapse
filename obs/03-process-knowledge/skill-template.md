# Synapse Skill 标准模板

> 所有新建 Skill 的 SKILL.md 必须按此模板编写。
> QA 门禁（/qa-gate）会检查强制字段是否存在，缺失即不合格。

---

## Frontmatter（YAML 头部，强制）

```yaml
---
name: skill-name              # Skill 唯一标识，与目录名一致
description: |
  一行中文说明。
  一行英文说明（供搜索和 Skill 列表展示）。
allowed-tools:                # 允许该 Skill 调用的工具白名单
  - Bash
  - Read
  - Write
  - Edit
argument-hint: "[参数提示]"   # 用户输入时的参数提示
---
```

---

## 正文结构（建议）

```markdown
# /skill-name — 功能标题

**执行团队：[团队名]（[specialist_id]）**

一段话描述 Skill 用途和触发场景。

---

## Step N: 步骤名称

**[specialist_id] 执行：**

步骤说明...

**GATE：关键写入操作必须有门禁验证。**

---

## 降级策略（Graceful Degradation）

| 依赖 | 不可用时的处理 |
|------|---------------|
| ... | ... |

---

## 执行约束

- 约束条款...
```

## 测试场景（强制，交付前必须定义）

> **强制字段** — 每个 Skill 必须定义至少 **1 个 Golden Path** + **1 个 Edge Case**。
> QA 门禁（/qa-gate）会按此场景执行冒烟测试，缺失本节视为 Skill 不合格。

### test_scenarios

#### Golden Path（必须，至少 1 个）

- **场景名称**：[正常使用场景的一句话描述]
- **输入**：`/skill-name [参数]`
- **前置条件**：[环境状态，如"personal_tasks.yaml 已存在且 inbox 有条目"]
- **预期结果**：
  - [ ] 文件变更：`[目标文件路径]`
  - [ ] 文件内容匹配：`[正则表达式或关键字]`
  - [ ] 工具调用链：`Read -> Edit -> Bash(git diff) -> Bash(git commit)`
  - [ ] git commit 消息包含：`[关键字]`
  - [ ] 最终输出包含：`[用户可见的确认信息]`

#### Edge Cases（必须，至少 1 个）

- **场景名称**：[边界/异常场景描述]
- **输入**：`/skill-name [特殊参数或无参数]`
- **前置条件**：[特殊环境状态，如"目标文件不存在"/"MCP 服务不可用"]
- **预期结果**：
  - [ ] [降级行为描述：如"跳过该步骤并在输出中标注不可用"]
  - [ ] [不应发生的行为：如"不应中断整体流程"/"不应产生空 commit"]

## 测试场景编写指南

### 什么是好的 Golden Path

- 覆盖 Skill 的**主要用途**（用户最常用的调用方式）
- 预期结果具体可验证（文件路径、内容正则、commit 消息）
- 工具调用链完整，能追踪执行流程

### 什么是好的 Edge Case

- 覆盖**降级策略**中列出的每种不可用场景
- 覆盖**空数据**场景（首次使用、文件为空、无匹配结果）
- 覆盖**参数异常**（无参数、格式错误的参数）
- 覆盖**并发/重入**（Skill 被连续调用两次会怎样）

### 数量建议

| Skill 复杂度 | Golden Path | Edge Cases | 合计 |
|--------------|-------------|------------|------|
| S级（简单）   | 1           | 1          | 2    |
| M级（常规）   | 1-2         | 2-3        | 3-5  |
| L级（复杂）   | 2-3         | 3-5        | 5-8  |
