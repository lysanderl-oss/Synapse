2026年4月10日，我和我的AI CEO "Lysander" 进行了一场长达数小时的对话。对话结束时，我们从零搭建了一套包含10个专业团队、49个AI Agent的完整运营体系。

这不是概念验证，不是Demo。这是一个正在持续运行的生产系统 — 每天自动生成AI情报日报，自动评估和执行行动建议，每周自动审计全员能力评分。

本文记录这个过程中最有价值的实践经验。

什么是 Harness Engineering

Martin Fowler 和 Red Hat 在2026年初确认了一个行业共识：AI Agent 的可靠性不取决于模型本身，而取决于你为它构建的"驾驭系统"。

Agent = Model + Harness

Model（模型）：Claude、GPT、Gemini — 人人可用，不是壁垒。

Harness（驾驭系统）：环境、约束、反馈、组织 — 这才是让AI可靠工作的关键。

我的理解是：模型就像一个极其聪明但没有纪律的天才。Harness就是给这个天才设定的工作规范、汇报流程、质量标准和绩效考核。

实践一：Guides + Sensors 双保险

Harness有两个核心机制：

Guides（前馈控制）：在AI行动之前，引导它做正确的事。

在我的系统中，CLAUDE.md就是最重要的Guide — 它定义了角色、流程、决策规则。AI每次启动都会读取这个文件，就像员工每天上班先看工作手册。

Sensors（反馈控制）：在AI行动之后，检查它是否做对了。

我在实践中发现了一个关键教训 — 光有Guides不够。我的AI CEO反复跳过"团队派单"环节，自己一个人干完所有活。规则写在文档里，但没有机制检查它是否遵守了。

解决方案：我加入了"强制团队派单制度" — AI在执行任何工作之前，必须先输出"谁做什么"的派单表。这不是建议，是强制检查点。同时增加了"执行审计师"角色，在事后验证派单是否存在。

这就是Harness Engineering的精髓：体系不靠"下次注意"，靠制度强制。

实践二：四级决策体系

我发现一个反直觉的设计：CEO不应该排在专家前面做决策。

最初的设计是：L1自动 → L2 CEO → L3专家 → L4总裁

我纠正为：L1自动 → L2专家 → L3 CEO → L4总裁

逻辑很简单：专业问题应该先过专家，管理者再基于专家建议做决策。CEO拍脑袋不如专家分析。这个原则同样适用于AI系统 — 让AI管理者基于AI专家的分析来决策，而不是直接判断。

实践三：Agent 也需要HR管理

这是最出人意料的发现。

我的49个AI Agent中，有8个评分为0分 — 它们被创建时没有任何能力标准，甚至连基本的档案格式都不统一。

于是我建立了完整的AI人力资源管理体系：

  • 强制Schema：每个Agent必须有标准化的档案，缺项不得上岗
  • 能力分级：A/B/C三级评定，C级（只写"项目管理"这样的空泛描述）不合格
  • 入职审批：HR Director 8项检查清单
  • 定期审计：每周一自动运行全员评分
  • 进化机制：当行业出现新技术时，自动更新相关Agent的能力描述

结果：全员平均分从64.1提升到93.8，不合格人数从11降到0。

实践四：情报闭环 — AI自己学习和进化

最让我满意的设计是"情报闭环管线"：

每天8am：AI自动搜索行业前沿 → 筛选3-5条有价值的发现 → 生成情报日报
每天10am：AI自动评估日报建议 → 4专家评分 → 批准后自动执行 → 生成成果报告

这意味着这个AI团队每天都在学习新技术、评估是否有用、然后自己动手落地。我只在Slack上收到最终结果通知。

给同行的建议

  1. 先定义角色，再谈技术。我的第一步是明确"谁是老板、谁是CEO、谁是执行者"，而不是讨论用什么模型。

  2. 不要相信"AI会自觉遵守规则"。一定要有Sensor（反馈机制），不能只靠Guide（前馈指令）。

  3. 专家先于管理者。无论是人类组织还是AI组织，专业判断应该排在管理决策前面。

  4. Agent的质量标准比数量重要。49个90分的Agent远比100个60分的Agent有价值。

  5. 让AI团队自己进化。不要手动管理每个Agent的能力更新，建立自动化的情报→评估→执行闭环。
