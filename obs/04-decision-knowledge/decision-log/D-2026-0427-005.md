- **议题**：Synapse Platform Phase 1 P0 纠偏 + Owner Input 替代策略 + 范围锁定
- **决策级别**：L3（专家评审团 + Lysander 决策；不上报总裁）
- **决策者**：Lysander CEO（基于专家评审团 5/5 一致建议）
- **日期**：2026-04-27
- **状态**：decided

## 背景与上下文

D-2026-0427-004 批准了 Synapse Digital Twin Collaboration Platform 战略方案。在产品设计规格 5 份文档与项目脚手架代码完成后，专家评审团进行第二轮综合评审（"做正确的事 / 正确的做事 / 把事做正确"三层评估），发现 4 项 P0 级设计/实现失配：

1. **状态机 SSOT 不一致**：02-data-model.md 文档定义 10 态与 src/core/fsm-engine.ts 实现 10 态名称几乎完全不重叠，下游 UX 文档与验收用例全部对应失败。
2. **Approval Gate 治理覆盖缺口**：approval-matrix.yaml 仅强制 awaiting_approval → completed 人审，但 UX 中 weekly_report 走 awaiting_review → completed 也是批准动作，未受治；case-types.yaml 的 requires_approval 字段语义与文档相悖；ApprovalGate 加载 matrix 失败时静默 fallback。
3. **Owner Input 路径需作废**：总裁本轮指令明确 Owner Input 不再由总裁个人提供，改由智囊团 + PMO + 行业最佳实践产出；但代码与 playbook 仍引用 owner-input-capture.md，D-2026-0427-004 前置条件 #1 被推翻但未在制品中反映。
4. **Phase 1 范围在文档与代码中数量不一致**：D-2026-0427-004 称"9 场景"，04-phase1-agent-specs.md 列 9 类，case-types.yaml 列 7 类（且名称不同），PM Agent 实际只实现 2 类。

完整评审报告：`obs/04-decision-knowledge/2026-04-27-synapse-platform-phase1-review.md`（评审分数 78/100）。

触发事件：总裁要求专家团对 Synapse Platform Phase 1 进行第二轮综合评审，授权专家团直接决策，不上报。

## 选项分析

### 选项 A：忽略失配，按原 Sprint A WP1→WP2→WP3→WP4 推进
- **优势**：时间表不变，对外形象稳定。
- **劣势**：状态机不一致会导致 UX 消息层与代码层对接全部返工；治理漏洞遗留；Owner Input 依赖未消解会卡住 PM Agent 实际跑通。
- **风险**：返工成本指数上升；Sprint B/C 时积累的债务可能压垮整个 Phase 1。

### 选项 B：插入 WP1.5 — 一致性回滚（Consistency Recovery），P0 全部修完后再进 WP2
- **优势**：建立 SSOT，治理硬规则真正不可绕过，Owner Input 替代策略落地，范围明确化。
- **劣势**：Sprint A 总周期增加 2-3 个工作日。
- **风险**：低；纯属技术债清理 + 文档对齐。

### 选项 C：推翻 5 份产品设计文档，以代码为唯一 SSOT，文档全部重写
- **优势**：代码即真理。
- **劣势**：5 份文档共 1700+ 行，丢失大量 UX 与验收设计；过度反应。
- **风险**：知识资产损失。

## 专家评审意见

**评审团 5/5 一致同意选项 B。** 详细评审与签字见报告文件。

关键论据：
- Case-driven 哲学、ApprovalGate 双重校验、TS 严格模式 — 这些"正确做的事"必须保留
- 失配项均为可定向修复的局部问题，无需推翻全局设计
- WP1.5 工作量约 2-3 Sprint Day，远小于 Sprint B/C 阶段返工成本
- Owner Input 替代策略被独立评估为"可行"，PMI/PRINCE2/Atlassian/Asana 的方法论足以支撑业内默认 + UserDossier 反向学习

## 决策结果

### 选择：选项 B + 配套指令

#### B-1 插入 WP1.5 — 一致性回滚（独立工作包）

**范围：**
1. **P0-1 状态机 SSOT 对齐**：以 fsm-engine.ts 当前实现（INTAKE/TRIAGED/PLANNED/IN_PROGRESS/AWAITING_REVIEW/AWAITING_APPROVAL/BLOCKED/COMPLETED/ARCHIVED/CANCELLED）为 SSOT，重写 docs/product/01-ux-interaction-design.md + 02-data-model.md 状态枚举与状态机表 + 05-acceptance-criteria.md 用例期望值。
2. **P0-2 Approval Gate 漏洞修复**：
   - approval-matrix.yaml 增加 `awaiting_review → completed` 强制 human + approvalRecord
   - case-types.yaml 引入 `review_required` 字段与 `requires_approval` 区分
   - approval-gate.ts loadMatrix 异常路径改为 fail-loud（throw 而非 warn fallback）
3. **P0-3 Owner Input 替代落地**（详 B-2）
4. **P0-4 Phase 1 范围锁定**：Sprint A 实际交付场景 = weekly_report + meeting_prep + service_request；其余 meeting_minutes / action_followup / decision_doc 推迟 Phase 1.5；retro_synthesis / dependency_notice / tbd_01 推迟 Phase 1.5+。
5. **测试矩阵补齐**（含修复 case-flow.test.ts L47 regex bug `/^CASE-d{8}-d{4}$/` → `/^CASE-\d{8}-\d{4}$/`）

**责任：** harness_engineer（主笔代码与配置）+ product_strategist（主笔文档）+ knowledge_engineer（PMI/PRINCE2 知识注入）+ integration_qa（测试矩阵）

**完成标志：** WP1.5 验收 = (a) 文档与代码状态机一致；(b) GOV-01~04 全部自动化测试通过；(c) PM Agent 在无 owner-input-capture.md 情况下可独立产出周报草稿。

#### B-2 Owner Input 替代策略（正式批准实施）

**知识源（注入 PM Agent system prompt 层 1 + playbooks）：**
- PMI PMBOK Guide 7th Edition — Performance Domains
- PRINCE2 2017 — Managing a Stage Boundary / End Stage Report
- Atlassian Status Reporting Playbook（5 段式：Status / Achievements / Risks / Decisions / Next Steps）
- Asana Project Status Reporting Best Practices（RAG + 自动数据接入）
- Synapse 内部：weekly-audit 6 维度、Decision Log 4 级、CLAUDE.md Harness 工作原则、_template.md 决策模板

**PMO 通用语言：** RAID 矩阵、RAG 状态、SMART 行动项、DACI 决策角色、Risk = Probability × Impact、SLO/SLA escalation。

**个性化路径：** UserDossier 反向学习。前 2-4 周通过 awaiting_review 用户修改样本积累 Dossier；PM Agent 二次同类 case 时引用偏好。

**作废处理：** agents/pm-agent/owner-input-capture.md 标注 DEPRECATED，保留为历史；不再被 playbook 引用。

#### B-3 Phase 1 不引入 LangGraph（追认 ADR）

D-2026-0427-004 战略文档提及"Claude Agent SDK + LangGraph"，但 package.json 未引入 LangGraph，case-driven FSM 已是显式编排。本决策正式追认 Phase 1 不使用 LangGraph，避免重复抽象。Phase 2 视复杂度再评估。

### 理由

- 本平台战略选择正确（case-driven、Slack 入口、Claude API、Approval Gate 治理）—— 必须保留
- 失配项均可定向修复，工作量可控
- WP1.5 是廉价的债务偿还，能避免 Sprint B/C 雪崩式返工
- Owner Input 替代策略已论证可行，去除总裁单点依赖也提升平台普适性

### 预期结果

- WP1.5 完成后，Phase 1 文档与代码一致，治理硬规则不可绕过
- PM Agent 可独立产出周报与会议材料，水准对齐 PMI/Atlassian 行业标准
- Sprint A 真实交付 3 个场景全链路（weekly_report / meeting_prep / service_request），含完整 GOV-01~04 治理验收
- 验收门槛：`obs/04-decision-knowledge/2026-04-27-synapse-platform-phase1-review.md` 中所列 P0 全部清零，评审分数复测 ≥ 90/100
