## Context

Phase 2 Sprint C placed `CREATE EXTENSION IF NOT EXISTS vector;` in `001_create_tables.sql` as a zero-cost stub. No vector columns exist. No embedding pipeline exists. Phase 3 Sprint E activates this.

The business case for RAG in Synapse is specific: the PM Agent (and ResearchAgent/ContentAgent) currently generates responses based solely on the case context and its hardcoded expert knowledge base. It cannot retrieve:
- Prior cases similar to the current intake request (semantic similarity)
- Relevant passages from the operator-curated knowledge base files
- Past agent reasoning artifacts that may inform the current decision

The corpus already exists at Phase 3 start:
- `agents/pm-agent/expert-knowledge-base.md` (372 lines)
- `agents/pm-agent/playbooks/weekly-report.md`, `meeting-prep.md`, `service-request.md`
- `agents/research-agent/playbooks/research-request.md`, `data-analysis.md`
- `agents/content-agent/playbooks/content-draft.md`, `decision-brief.md`
- All persisted Artifacts from 161 test runs (in runtime-data/artifacts/)
- All persisted EvidenceBundles

The question is: what embedding model, what chunking strategy, what schema changes, and what retrieval pattern best fits this codebase and use case?

---

## Options Evaluated

### Option A — OpenAI text-embedding-3-small (recommended)
1536-dimensional embeddings. Cost: ~$0.02 per million tokens. Latency: ~50ms per API call. Already in the project's dependency ecosystem (project uses Claude for generation; OpenAI embeddings are the industry standard for pgvector and have first-class node-postgres support).

### Option B — Anthropic voyage-3-lite
768-dimensional. Anthropic-native embeddings. No OpenAI dependency needed.
- Cons: Requires adding @anthropic-ai/sdk voyage integration; less community tooling for pgvector with voyage; pgvector index tuning documented primarily for 1536-dim models. For a codebase already using Claude for generation, adding voyage adds a second Anthropic sub-product; the simpler choice for pgvector operational experience is text-embedding-3-small.

### Option C — Local embedding model (e.g., all-MiniLM-L6-v2 via onnx)
No external API cost. 384-dimensional.
- Cons: Requires bundling an ONNX runtime; significantly more complex CI setup; 384-dim models underperform on domain-specific professional text (PM reports, research requests, decision briefs). Not appropriate for production quality.

---

## Rationale for Option A

**Embedding model: text-embedding-3-small (1536-dim)**
- Industry standard for pgvector deployments
- Balances quality, cost, and latency
- pgvector HNSW index performs well at 1536-dim
- openai npm package is well-supported in Node.js ecosystem

**Schema additions (in a new migration `003_add_embeddings.sql`):**
```sql
-- Add embedding columns to relevant tables
ALTER TABLE cases ADD COLUMN IF NOT EXISTS embedding vector(1536);
ALTER TABLE agent_memory ADD COLUMN IF NOT EXISTS embedding vector(1536);
ALTER TABLE artifacts ADD COLUMN IF NOT EXISTS embedding vector(1536);

-- HNSW index for approximate nearest-neighbor search
CREATE INDEX IF NOT EXISTS cases_embedding_hnsw
  ON cases USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);

CREATE INDEX IF NOT EXISTS artifacts_embedding_hnsw
  ON artifacts USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);
```

**Chunking strategy:**
- Knowledge base files (expert-knowledge-base.md, playbooks): chunk by markdown section (H2/H3 boundary). Max 512 tokens per chunk with 64-token overlap.
- Case artifacts (Artifact.content): chunk by paragraph. Max 256 tokens per chunk (artifacts are already concise).
- Agent memory (AgentMemory): embed as single unit (typically <200 tokens).

**Retrieval pattern (Retrieval-Augmented Generation for PM Agent):**
At case intake (INTAKE → TRIAGED transition), before calling Claude API:
1. Embed the case description (requestText).
2. Query cases table: `SELECT id, data FROM cases WHERE embedding IS NOT NULL ORDER BY embedding <=> $1 LIMIT 5`
3. Query artifacts table: `SELECT id, data FROM artifacts WHERE embedding IS NOT NULL ORDER BY embedding <=> $1 LIMIT 3`
4. Inject retrieved context into agent system prompt as a "relevant prior cases" section.

This is a minimal but production-worthy RAG pattern. No langchain, no vector abstraction layer — direct pgvector SQL queries. Keeps the codebase's "no magic frameworks" philosophy.

**Corpus ingestion pipeline (`src/rag/corpus-ingester.ts`):**
- Reads all files from `agents/*/expert-knowledge-base.md`, `agents/*/playbooks/*.md`
- Chunks each file per above strategy
- Calls OpenAI embeddings API in batches of 100
- Upserts into a new `knowledge_chunks` table:
  ```sql
  CREATE TABLE IF NOT EXISTS knowledge_chunks (
    id TEXT PRIMARY KEY,
    source_file TEXT NOT NULL,
    chunk_index INT NOT NULL,
    content TEXT NOT NULL,
    embedding vector(1536),
    created_at TIMESTAMPTZ DEFAULT now()
  );
  ```
- Script: `npm run rag:ingest` — idempotent, safe to re-run

**Automatic case embedding (`src/rag/case-embedder.ts`):**
- Hook into CaseService: after an Artifact is saved (AGENT_COMPLETED event), compute and store its embedding
- Hook into case creation: embed case.requestText and store in cases.embedding
