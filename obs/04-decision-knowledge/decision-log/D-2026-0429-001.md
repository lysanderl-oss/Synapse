## 三、根因分析

### 根因分类：**F — Credential 权限丢失**

**根本原因**：`查询注册表` 节点引用的 Notion API Credential（`id: qdn9QguMUFJGJrGb`，名称 `Notion API`）权限已失效或无法被当前工作流访问。n8n 在执行时检测到节点无法访问该 Credential，在 6-9ms 内立即抛出权限错误，工作流从未进入实际业务逻辑。

**可能的子原因（需逐一排查）：**

| 优先级 | 子原因 | 判断依据 |
|-------|-------|---------|
| 高 | Credential 被删除或迁移 | 错误描述 "please recreate" 是 n8n 删除/迁移场景下的标准报错 |
| 高 | Credential 所有者变更，未共享给当前工作流所在项目 | WF-01 属于 personal project `dRj3b1ZnqLhLX5sX`，若 Credential 是从其他 project 迁移来的则会出现此错误 |
| 中 | Notion Integration Token 已过期或被 Revoke | Token 层面失效后 n8n 也可能将 Credential 标记为不可用 |

**注意**：Schedule Trigger 每 5 分钟触发属于**正常设计**，不是异常，不需修改。5 分钟轮询 + "无新项目则静默结束"是合理的低耦合轮询架构。

**pmo-api 状态**：健康，`{"status":"ok","wbs_script":"/app/wbs_to_asana.py","wbs_exists":true,"hub_exists":true}`，不是本次错误的原因。

## 四、处理方案

### 根因

`查询注册表` 节点所引用的 Notion API Credential（ID: `qdn9QguMUFJGJrGb`）被 n8n 判定为无访问权限，导致工作流在触发后 7ms 内立即失败，业务逻辑从未执行。

### 影响范围

- **功能影响**：WF-01 项目自动初始化完全不可用，所有 Notion 注册表中待初始化的新签约项目均无法触发 PMO 流程
- **资源消耗**：每 5 分钟产生一次错误执行记录，累计大量 error 日志（当前已有 168-172 共 5+ 次，可能持续更长时间）
- **数据影响**：无数据损坏，因为工作流从未进入写入阶段

### 处理步骤

1. **登录 n8n.janusd.io → Credentials** 页面，搜索 `Notion API` 或 `qdn9QguMUFJGJrGb`，确认该 Credential 是否存在

2. **如果 Credential 已删除**：
   - 新建一个 Notion API Credential（类型: `notionApi`）
   - 填入有效的 Notion Integration Token（从 notion.so/my-integrations 获取）
   - 确认该 Integration 已被添加到目标数据库 `61e17074-706d-4e17-855d-f34b15d6a75c`（Notion 注册表）的 Connections 中
   - 进入 WF-01 工作流，将所有引用 `Notion API` 的节点（`查询注册表`、`标记为初始化中`、`回填注册表链接` 等共 7 个 httpRequest 节点）重新绑定到新 Credential

3. **如果 Credential 存在但权限不足（项目隔离）**：
   - 进入 Credential 详情页 → Share → 将 Credential 共享给 WF-01 所在项目（personal project `dRj3b1ZnqLhLX5sX`）

4. **如果 Notion Token 已过期**：
   - 前往 notion.so/my-integrations，找到对应 Integration
   - 更新或重新生成 Token
   - 在 n8n Credential 中更新 Token 值

5. **修复后手动触发 WF-01 一次**，观察执行是否通过 `查询注册表` 节点

### 修复后验证

- 手动执行 WF-01，确认 `查询注册表` 节点返回 `{"results": [...]}` 或 `{"results": []}` 而非 credential 错误
- 在 Executions 页面确认最近一次执行 status = `success`（无新项目时走 `无新项目（结束）` 也是 success）
- 观察后续 5 分钟自动触发的执行均为 success

### 预防措施

1. **Credential 删除前检查引用**：n8n 删除 Credential 时应先查看"Used by"关联的工作流列表，避免盲目删除
2. **添加 Credential 有效性监控**：在 WF-01 或独立 Health Check WF 中，定期测试 Notion API 连接，发现失败时 Slack 告警，不依赖执行日志人工发现
3. **错误触发 Slack 通知**：WF-01 settings 中开启 Error Workflow，当 status=error 时自动通知总裁或 PMO 频道
