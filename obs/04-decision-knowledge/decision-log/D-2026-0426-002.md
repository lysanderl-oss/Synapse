# D-2026-0426-002：管线产品（Pipeline Product）治理元规则

## Decision

lysander.bond 进入"管线产品"治理框架。所有未来变更基于版本基线管理，三层版本（MAJOR/MINOR/PATCH）对应不同审批层级（L4/L3/L2），每版本必须 UAT + 度量快照。

## Context

D-2026-0426-001 落地后，lysander.bond 不再是"项目"（一次交付完即结束），而成为长期运营的对外品牌阵地。需要建立元规则：什么变更走什么审批、版本如何管理、变更如何分类、UAT 如何跑、度量如何快照。否则未来变更会走回"无审批 / 无基线 / 无回滚"的混乱状态。

## Options Considered

- A：管线产品框架（三层版本 + 三档审批 + 三批次执行）
- B：保持项目模式，每次变更单独走审批流程
- C：完全自动化（无审批，靠 CI 拦截）

## Selected

**A：管线产品框架**

三层版本约定：
- **MAJOR**（v2.0+）：架构 / 品牌转向 / 商业模式 → L4 总裁审批，三轮 UAT
- **MINOR**（v1.2, v1.3）：战略内容 / IA / 双语扩展 → L3 Lysander 决策，双轮 UAT
- **PATCH**（v1.1.1）：Bug / 文案 / 链接 / SEO → L2 Lysander 自主，仅质量验证

变更分类标签（每个 commit/PR 必须带）：
`strategic` / `feature` / `fix` / `chore` / `docs` / `i18n`

## Consequences

- Positive：变更可审计、可回滚、可度量；审批负担按风险分层；总裁仅看 MAJOR
- Negative：开发流程多一层版本管理负担
- Reversibility：易（如发现 PATCH 比例过低或 MAJOR 误用，可调整三档定义）

- 首次 MAJOR 候选：v2.0 战略改造完成后
