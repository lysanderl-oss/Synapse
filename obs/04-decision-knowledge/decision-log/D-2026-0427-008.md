# D-2026-0427-008 — PostgreSQL Migration Strategy (Adapter Pattern)

## Context

Phase 1 uses `FileStore<T>` as the generic persistence layer. Each domain store (`CaseStore`, `ArtifactStore`, `AuditStore`, `EvidenceBundleStore`) wraps `FileStore` directly — it is not injected through an interface. The migration strategy must be chosen now so Sprint C can be executed cleanly.

**Current FileStore API:**
```
get(id): Promise<T | null>
save(entity): Promise<T>
delete(id): Promise<boolean>
list(): Promise<T[]>
find(predicate): Promise<T[]>
findOne(predicate): Promise<T | null>
count(): Promise<number>
```

The `find(predicate)` method uses in-memory full-scan. At Phase 2 scale (hundreds of cases, thousands of audit events) this is acceptable but would degrade at Phase 3 scale.

---

## Options Evaluated

**Option A — Full replacement:** Remove FileStore entirely, rewrite all stores to use pg/postgres.js directly.
- Risk: Big-bang change; breaks tests; can't A/B or roll back.
- Verdict: Rejected. High risk for no added benefit vs. Option B.

**Option B — Adapter pattern (recommended):** Extract `IStore<T>` interface matching the current FileStore API. FileStore implements it. PgStore implements it. Each domain store takes `IStore<T>` in its constructor (dependency injection). FileStore remains default; PgStore is activated via env flag `SYNAPSE_STORE_BACKEND=postgres`.
- Risk: Low. No behavioral changes. Tests continue using FileStore. Migration can be toggled per environment.
- Migration path: `scripts/migrate-to-postgres.ts` reads all JSON files from `runtime-data/` and inserts into Postgres via PgStore.
- Rollback: Set `SYNAPSE_STORE_BACKEND=file` to revert to file storage.
- Verdict: Recommended.

**Option C — Dual-write:** Write to both FileStore and PgStore simultaneously during transition period.
- Risk: Increased complexity; two sources of truth; sync bugs.
- Verdict: Rejected. Adds complexity without benefit given the single-server deployment model.

---

## Verdict

**APPROVED: Option B — Adapter pattern.**

Implementation plan:
1. Extract `IStore<T>` interface in `src/storage/store-interface.ts`
2. FileStore implements `IStore<T>` (no behavioral change)
3. PgStore implements `IStore<T>` using `postgres` npm package
4. All domain stores accept `IStore<T>` constructor injection; default remains FileStore
5. `src/utils/store-factory.ts` reads `SYNAPSE_STORE_BACKEND` env and returns correct implementation
6. `scripts/migrate-to-postgres.ts` one-way migration tool (with dry-run flag)
7. Down migration: `scripts/rollback-to-file.ts` exports Postgres records back to JSON files
8. Schema: each entity type gets its own table; audit_events is append-only with `BRIN` index on `created_at`

**pgvector decision:** pgvector extension install is included in the migration schema DDL as a no-op stub, but no RAG features are implemented in Phase 2. This allows Phase 3 to enable semantic search without a schema migration.

---

## Acceptance Criteria for Sprint C

- `SYNAPSE_STORE_BACKEND=file` (default): all 83+ Phase 2 tests pass unchanged
- `SYNAPSE_STORE_BACKEND=postgres`: all 83+ tests pass against a real Postgres instance (vitest --environment node + testcontainer or local Postgres)
- Down migration script produces JSON files identical to pre-migration state (verified by checksum)
- `tsc 0 errors` maintained
- Case creation latency with Postgres: ≤ 50ms p99 on local hardware
