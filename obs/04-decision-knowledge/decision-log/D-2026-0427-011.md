## Context

Phase 2 delivered at 93/100, with all 7 scenarios live, IStore<T> adapter pattern in place, PgStore implemented, and pgvector extension stub in schema DDL. Three structural gaps were explicitly deferred to Phase 3:

1. Domain stores still compose FileStore internally — the STORAGE_BACKEND switch in index.ts creates a Pool but then passes new CaseStore() (FileStore-backed) regardless. PgStore is implemented but never wired into domain stores.
2. PgStore.find() / findOne() fetch all rows and filter in JS — no server-side WHERE predicate.
3. PgStore tests use mocked pg.Pool — no live database integration tests exist.

Additionally, D-2026-0427-007 deferred pgvector RAG, RBAC, web dashboard, SLA Slack alerting, and real Socket Mode verification to Phase 3.

The Phase 3 scope question is: which of these items deliver enough business value to justify the build cost, and in what sequence?

---

## Options Evaluated

### Option A — Infrastructure-heavy Phase 3
Prioritize: DI refactor → live DB tests → pgvector RAG → RBAC → dashboard.
- Risk: RAG and RBAC in the same phase creates a very large scope. RBAC has no driving user story. Dashboard has no driving user story. Over-builds.

### Option B — RAG-first Phase 3
Skip DI refactor, jump straight to pgvector RAG (add vector columns to PgStore tables, build embedding pipeline).
- Risk: Builds on a broken foundation — domain stores still use FileStore even when STORAGE_BACKEND=postgres. RAG over a live DB that the app isn't actually writing to is incoherent. Technically unsound.

### Option C — Foundation-first then RAG (recommended)
Sprint D: Complete domain store DI wiring + server-side queries + live DB integration tests (the actual "Phase 2 missing half").
Sprint E: pgvector RAG pipeline (corpus ingestion + semantic retrieval for PM Agent context enrichment).
Sprint F: Real Slack Socket Mode verification + SLA Slack alerting. RBAC Phase 4.
Sprint G (conditional): Web dashboard — only if a concrete user story emerges; otherwise Phase 4.

---

## Rationale

**DI refactor must be first.** The Sprint C report explicitly documents: "direct PgStore<T> injection into domain stores is the Phase 3 refactor." The STORAGE_BACKEND=postgres path today creates a Pool object then discards it — every domain store uses FileStore regardless of the env var setting. This means the PostgreSQL migration that Phase 2 was supposed to deliver is actually not functional end-to-end. Sprint D closes this gap before any higher-layer feature is built on top.

**Server-side queries must accompany DI refactor.** Once domain stores accept IStore<T> via injection, PgStore.find() being a full-table-scan is no longer acceptable for audit logs (which grow unboundedly). Typed query methods (findByStatus, findByCaseId, findByEventType, findAfter) must be added to PgStore and to CaseStore / AuditStore before Sprint E loads data at scale.

**Live DB tests are a Phase 3 gate.** Phase 2 accepted PgStore tests with mocked pg.Pool as a known gap. Phase 3 cannot ship PgStore as a real production backend without testcontainer-based integration tests.

**pgvector RAG is correctly Phase 3 if and only if Sprint D ships first.** With a real Postgres backend wired end-to-end, adding vector columns and a semantic search pipeline is coherent. The PM Agent's expert-knowledge-base.md (372 lines) and the 7 scenario playbooks form the initial corpus. The business value is concrete: PM Agent context enrichment at case intake reduces hallucinated responses and anchors output quality to the operator's actual knowledge base.

**RBAC has no Phase 3 user story.** The platform remains single-tenant (Synapse-PJ). No new organizations are onboarding in the Phase 3 window. RBAC for a single organization adds complexity with zero end-user benefit. Defer to Phase 4 with a clear trigger condition: first multi-workspace deployment.

**Web dashboard has no Phase 3 user story.** Current observability via Slack thread + Notion is sufficient. No specific user is blocked by the absence of a dashboard in Phase 3. Defer to Phase 4, triggered by the first external-facing demo or stakeholder reporting requirement.

**Real Slack Socket Mode and SLA alerting are Phase 3 because they close real production gaps.** The current Bolt smoke test verifies listener registration against a nock shim. There is no verification that real socket-mode connection, reconnect, and payload dispatch work end-to-end. SLAMonitor correctly marks breaches in the audit log and case record but has no Slack notification path — the business requirement is for the PM Agent owner (Lysander Liu) to receive a Slack DM when an SLA is breached.

**APPROVED: Option C — Foundation-first then RAG.**
