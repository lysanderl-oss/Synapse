# D-2026-0428-017 — Web Dashboard vs Slack-Native Reporting

- **Date:** 2026-04-28
- **Decision Level:** L3 (Lysander + Expert Panel)
- **Status:** APPROVED
- **Decided by:** Expert Panel (product_strategist · think_tank · harness_engineer)
- **Related:** D-2026-0427-011 (Phase 3 Won't — web dashboard) · D-2026-0428-016 (Phase 4 scope)

---

## Context

The Phase 3 scope decision (D-011) deferred a web dashboard to Phase 4 with the trigger condition: "first external-facing demo or stakeholder reporting requirement requiring non-Slack visibility." The Phase 4 review panel must decide: (a) has the trigger condition been met, and (b) if a reporting surface is built, which implementation approach?

---

## Trigger Condition Assessment

**Has the trigger condition been met?**

The D-011 trigger was: "first external-facing demo or stakeholder reporting requirement requiring non-Slack visibility."

Current state:
- The platform is single-tenant (Synapse-PJ internal)
- No external parties have been onboarded or are scheduled to demo
- The only known reporting need is case lifecycle visibility for the internal PM workflow

**Verdict: Trigger condition NOT met.** A full web dashboard is not warranted in Phase 4 under the D-011 binding conditions.

However, there IS a legitimate internal reporting need: the president and PM Agent owner cannot easily see active case FSM states, artifact counts, or SLA status without searching audit logs. This is an operational friction point that can and should be addressed — but it does not require a new web infrastructure layer.

---

## Options Evaluated

### Option A — Next.js SPA Dashboard
Full web application with case table, FSM state viewer, agent performance metrics, audit trail viewer.

**Cost:** XL. Requires: Next.js setup, authentication layer (or open dev access), API routes, React components, hosting/serving infrastructure. Minimum 2 sprints.

**Value:** High — but only if external stakeholders need visibility. For internal single-tenant use, the build cost is disproportionate to the operational benefit.

**Verdict:** Rejected for Phase 4. Revisit when D-011 trigger condition is met.

### Option B — Simple Express + EJS server-rendered dashboard
Lightweight Node.js web server reading from PgStore, rendering case table with EJS templates.

**Cost:** L. Requires: Express setup, EJS templates, basic CSS, auth (at minimum HTTP basic auth), serving configuration. 1.5 sprints.

**Value:** Moderate. Simpler than Next.js but still adds a new infrastructure component that needs maintenance.

**Verdict:** Rejected. Still disproportionate for current single-tenant internal use.

### Option C — Slack Canvas (native Slack, zero new infra)
Use Slack Canvas (collaborative document embedded in Slack workspace) to auto-publish case status summaries. The bot creates/updates a Canvas document in a dedicated `#case-status` channel.

**Cost:** M. Requires: Slack `canvases.create` / `canvases.edit` API calls, Canvas markdown formatting.

**Value:** Moderate. Provides persistent, formatted case status view within Slack. But Canvas is a relatively new Slack primitive, its API has rate limits, and it doesn't provide real-time querying — it's a document updated by the bot, not an interactive query interface.

**Verdict:** Considered but deprioritized. Canvas write API is immature; documents can become stale between bot updates.

### Option D — Slack-native slash command reporting (recommended)
Implement `/case-status [caseId]` slash command that queries PgStore and returns a structured Block Kit message with:
- Case FSM state, type, created-at, owner agent
- Artifact count and last artifact summary
- SLA status (on track / warning / breached, time remaining)
- Most recent audit log entry

Optionally, `/cases-active` returns a paginated list of all open cases.

**Cost:** S-M. Requires: command handler (already have `command-handler.ts`), PgStore query (already have server-side queries in Phase 3), Block Kit message builder (already have `message-builder.ts`).

**Value:** Directly solves the operational need (case visibility) using existing infrastructure. Zero new infrastructure. Uses the same Slack surface users already work in.

**Verdict:** Recommended. This is the correct Phase 4 choice.

---

## Verdict

**APPROVED: Option D — Slack-native slash command reporting. No web dashboard in Phase 4.**

### Implementation Contract

- **Command:** `/case-status [caseId]` — returns Block Kit message with full case state snapshot
- **Command:** `/cases-active` — returns paginated list of open cases (≤10 per response, with "Load more" action button)
- **Implementation file:** extend `src/integrations/slack/command-handler.ts`
- **Data source:** PgStore (or FileStore via IStore<T> adapter — same interface works for both backends)
- **No new dependencies required** — Block Kit builder and command handler already exist
- **Phase sprint assignment:** Sprint I (Should-Have in D-016)

### Web Dashboard Trigger Conditions (Binding)

A web dashboard moves to Must-Have when ANY of the following is met:
1. **External demo:** A demo for a prospective client, investor, or partner requires case visibility outside Slack
2. **Non-Slack stakeholder:** A named stakeholder with a reporting requirement cannot use Slack (e.g., board member, external PM)
3. **Compliance reporting:** An audit or compliance requirement mandates a documented case history report in a format other than Slack messages

Until then, Slack-native reporting is the production solution.
