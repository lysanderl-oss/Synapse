# D-2026-0427-012 — Domain Store DI Refactor Approach (Phase 3 Sprint D)

- **Date:** 2026-04-27
- **Decision Level:** L3 (Lysander + Expert Panel)
- **Status:** APPROVED
- **Decided by:** Expert Panel (harness_engineer · integration_qa · product_strategist)
- **Related:** D-2026-0427-008 (PG adapter, established IStore<T>) · D-2026-0427-011 (Phase 3 scope)

---

## Context

Phase 2 Sprint C implemented IStore<T> and PgStore<T>, but domain stores were not refactored to accept constructor injection. The Sprint C report explicitly states:

> "direct PgStore<T> injection into domain stores is the Phase 3 refactor as specified in D-2026-0427-008"

The current state in src/index.ts `createStores()`:
```typescript
// Both branches return FileStore-backed domain stores regardless of STORAGE_BACKEND
caseStore: new CaseStore(),   // internally constructs FileStore<Case>
auditStore: new AuditStore(), // internally constructs FileStore<AuditEvent>
evidenceBundleStore: new EvidenceBundleStore(), // internally constructs FileStore
```

The `Pool` object created in the postgres branch is referenced but never passed anywhere — it is dead code. The STORAGE_BACKEND=postgres path is therefore non-functional: the application writes to FileStore regardless of the env var setting.

All 6 domain store classes (CaseStore, AuditStore, EvidenceBundleStore, ArtifactStore, UserStore, AgentStore) follow the same pattern: they construct their own FileStore<T> internally in the constructor.

The question is: what is the correct refactor approach, and what changes are required?

---

## Options Evaluated

### Option A — Thin constructor injection (recommended)
Add an optional `store?: IStore<T>` parameter to each domain store constructor. When omitted, behavior is identical to Phase 2 (constructs FileStore internally). When provided, uses the injected store. The StoreFactory in index.ts creates PgStore instances and injects them.

```typescript
// CaseStore example
export class CaseStore {
  private readonly store: IStore<Case>;
  constructor(store?: IStore<Case>) {
    this.store = store ?? new FileStore<Case>({ ... });
  }
}
```

- Pros: Zero behavioral change when store is omitted. All existing tests pass unchanged — they construct CaseStore() without arguments. PgStore injection path is testable via explicit injection.
- Cons: Optional parameter could be confusing; callers might not realize they need to inject for postgres mode.

### Option B — Required injection, factory creates all stores
Make the IStore<T> parameter required in domain store constructors. Require all callers to use StoreFactory.

```typescript
export class CaseStore {
  constructor(private readonly store: IStore<Case>) {}
}
// All tests must be updated to: new CaseStore(new FileStore<Case>(...))
```

- Pros: Explicit about the dependency. No hidden FileStore fallback.
- Cons: All 161 tests that construct domain stores directly must be updated. High churn for low marginal benefit — the tests are unit tests that don't need to parameterize storage.

### Option C — Separate PgCaseStore subclass
Create `PgCaseStore extends CaseStore` that overrides storage with PgStore, no constructor changes to CaseStore.

- Cons: Inheritance for dependency substitution is an anti-pattern. Doubles the store class count. Breaks Liskov in subtle ways (PgCaseStore constructor has pg.Pool dependency, CaseStore does not). Rejected.

---

## Rationale

Option A preserves all 161 existing tests with zero changes while enabling real PgStore injection. The optional parameter approach is clear: StoreFactory is the canonical construction path for production; direct construction (used in tests) always defaults to FileStore. This is the standard pattern for backward-compatible DI adoption in TypeScript.

The key StoreFactory change in src/index.ts is:
```typescript
function createStores(config, pool?: Pool) {
  if (pool) {
    return {
      caseStore: new CaseStore(new PgStore<Case>(pool, 'cases')),
      auditStore: new AuditStore(new PgStore<AuditEvent>(pool, 'audit_logs')),
      evidenceBundleStore: new EvidenceBundleStore(new PgStore<StoredEvidenceBundle>(pool, 'evidence_bundles')),
      artifactStore: new ArtifactStore(new PgStore<Artifact>(pool, 'artifacts')),
      userStore: new UserStore(
        new PgStore<UserProfile>(pool, 'user_profiles'),
        new PgStore<...>(pool, 'user_dossiers')
      ),
      agentStore: new AgentStore(
        new PgStore<...>(pool, 'agent_memory'),
        new PgStore<...>(pool, 'agent_roles')
      ),
    };
  }
  // FileStore path (default, unchanged behavior)
  return {
    caseStore: new CaseStore(),
    auditStore: new AuditStore(),
    // ...
  };
}
```

**Special cases requiring care:**
- `UserStore` composes two FileStore instances (profileStore + dossierStore). Its optional injection must accept two IStore<T> parameters.
- `AgentStore` composes two FileStore instances (memoryStore + roleStore). Same dual-injection pattern.
- `EvidenceBundleStore` uses a StoredEvidenceBundle adapter type for the `updatedAt` field. PgStore injection must use this internal type, not the public EvidenceBundle.
- `AuditStore.append()` must remain semantically append-only even with PgStore — the IStore<T>.save() upsert behavior is correct since AuditEvent IDs are UUID-generated and never collide.

---

## Verdict

**APPROVED: Option A — thin optional constructor injection.**

### Acceptance Criteria for Sprint D DI Refactor

1. All 6 domain store constructors accept an optional IStore<T> parameter (or two for dual-store classes).
2. When parameter omitted: behavior is byte-for-byte identical to Phase 2 (FileStore-backed).
3. All 161 existing tests pass without modification.
4. `STORAGE_BACKEND=postgres` with a real DATABASE_URL creates a Pool and passes PgStore instances to all domain stores via StoreFactory.
5. New test: `tests/integration/di-wiring.test.ts` — verify that with a testcontainer Postgres, save/get/list round-trips pass for all 6 domain stores via PgStore injection.
6. `tsc --noEmit`: 0 errors.

### Server-Side Query Methods (bundled with Sprint D)

PgStore must add the following server-side query methods to eliminate in-memory full-table scans for the highest-volume operations:

```typescript
// Additional methods on PgStore<T> (Sprint D)
findWhere(conditions: Record<string, unknown>): Promise<T[]>
// Generates: SELECT data FROM table WHERE data @> $1::jsonb
// JSONB containment — covers all CaseFilter field equalities
```

And domain stores must expose:
- `CaseStore.findByStatus(status)` → `WHERE data->>'status' = $1` (already on CaseStore but goes through find() on PgStore — redirect to WHERE clause)
- `AuditStore.query()` → push date-range and caseId filters to SQL WHERE (currently JS filter on full list())
- `EvidenceBundleStore.findByCaseId()` → `WHERE data->>'caseId' = $1`

Target: p99 latency ≤ 10ms for single-entity operations, ≤ 50ms for filtered list operations against a 10,000-row table on local hardware.
