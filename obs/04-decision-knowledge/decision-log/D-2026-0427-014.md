# D-2026-0427-014 — RBAC Timing Decision (Phase 3 or Phase 4?)

- **Decision Level:** L3 (Lysander + Expert Panel)
- **Status:** APPROVED
- **Decided by:** Expert Panel (product_strategist · think_tank · harness_engineer)

## Context

RBAC was explicitly deferred from Phase 2 in D-2026-0427-007 with the rationale: "single-tenant deployment; `authorized_tools` in AgentRole provides sufficient agent boundary." The Phase 2 acceptance report (93/100) reaffirmed this as item 6 in Phase 3 suggestions with the note "Phase 3 multi-tenant扩展时引入."

The Phase 3 review panel must now make a binding decision: is there a Phase 3 user story that justifies RBAC, or is it Phase 4?

---

## Analysis

### What "RBAC" would mean for this codebase

Human RBAC in Synapse would govern:
- Who can submit cases of which types (e.g., only designated users can submit `decision_brief`)
- Who can approve cases (currently handled by `approval-matrix.yaml` — already RBAC-like for approvals)
- Who can access case history (currently no multi-user access layer; all Slack users in the workspace can trigger the bot)
- Agent-level: which agents can invoke which tools (currently `authorized_tools` in AgentRole YAML)

The approval-matrix already provides the most critical access control: `require_human_approval` + `approval_group` per case type. This is functional RBAC for the approval workflow.

### Is there a Phase 3 user story?

**Current state:** Single Slack workspace. Single organizational tenant (Synapse-PJ). All Slack users who can message the bot can submit any case type. There is no requirement to restrict case submission by user role — any employee submitting a research_request or decision_brief is the intended usage.

**Scenario that would require RBAC:** A second organization (different Slack workspace or different approval group) is added. Or the president explicitly requires that only managers can submit decision_brief cases.

**Neither scenario is present in Phase 3.** The original requirements document (Section 2.2) lists "多语言和多角色协同" as a goal but does not define a specific RBAC user story. The current `approval-matrix.yaml` with `approval_group` already handles the role-based approval dimension.

### What RBAC would cost vs. deliver

**Cost:** M-L complexity. Requires:
- A roles table and user-role assignments
- Middleware in the Slack event handler to check submitter role before creating a case
- Role management commands (Slack slash commands or admin UI)
- Tests for all role-gated paths
- Integration with Slack user identity (userId → role lookup)

**Delivery value at Phase 3:** Zero. There is no user waiting for this. The platform already controls what agents can do (AgentRole.authorized_tools). Human access control for a single-tenant deployment where all users are trusted employees adds process overhead without security or business benefit.

---

## Options Evaluated

### Option A — Include RBAC in Phase 3
**Verdict:** Rejected. No driving user story. Would consume Sprint F-G bandwidth that is better spent on SLA alerting and Socket Mode verification — both of which close real production gaps. Building RBAC speculatively for multi-tenant is premature optimization.

### Option B — Defer RBAC to Phase 4 with binding trigger condition (recommended)
Phase 4 RBAC is triggered by the first multi-workspace deployment OR an explicit president requirement for case-type-level submission restrictions. Until then, the existing `authorized_tools` (agent RBAC) + `approval-matrix.yaml` (approval RBAC) are sufficient controls.

### Option C — Lightweight RBAC stub in Phase 3 (Slack workspace role check only)
Add a Slack workspace admin check: only workspace members with a specific Slack user group can submit `decision_brief` or `data_analysis` cases.
**Verdict:** Rejected. Even this "stub" requires Slack admin API integration and user-group management, which is M complexity. The business does not need it at Phase 3.

---

## Verdict

**APPROVED: Option B — RBAC is Phase 4, not Phase 3.**

### Binding Phase 4 Trigger Conditions

RBAC becomes a Phase 4 Must-Have when ANY of the following conditions is met:

1. **Multi-workspace:** A second Slack workspace is connected to the platform (first external tenant).
2. **Submission restriction request:** The president explicitly requires that case submission for any case type be restricted to a specific role/group.
3. **Audit requirement:** An external compliance or audit requirement mandates documented role-based access controls.

### What Phase 3 Does Instead

Phase 3 Sprint F (the slot where RBAC would have gone) is used for:
- Real Slack Socket Mode integration test (closes a production-confidence gap)
- SLA Slack alerting (closes a real operational gap — currently SLA breaches log to audit but no one is notified)
- Agent knowledge base output quality rubric gate for ResearchAgent / ContentAgent

This is a better use of the Phase 3 slot: it closes three real production gaps versus building one speculative access-control layer that serves no current user.
