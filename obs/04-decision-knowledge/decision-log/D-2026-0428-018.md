## Context

D-2026-0427-014 deferred RBAC to Phase 4 with three binding trigger conditions:
1. Multi-workspace: a second Slack workspace is connected
2. Submission restriction request: the president explicitly requires role-based case submission restriction
3. Audit requirement: external compliance mandate for documented RBAC controls

The Phase 4 review panel must assess whether any trigger condition is now met.

---

## Trigger Condition Assessment

### Trigger 1: Multi-workspace deployment
**Status: NOT MET.** The platform remains connected to a single Slack workspace (janussandbox.slack.com for testing, single production workspace for Synapse-PJ). No second organization or workspace is in the onboarding pipeline.

### Trigger 2: Submission restriction request
**Status: NOT MET.** No president directive has been received requiring that specific case types (e.g., `decision_brief`, `data_analysis`) be restricted to named users or roles. The current open-submission model (any workspace member can trigger the bot) remains operationally appropriate for a single-team deployment.

### Trigger 3: Audit or compliance requirement
**Status: NOT MET.** No external audit or compliance mandate has been raised requiring documented RBAC controls.

## Would RBAC Deliver Value in Phase 4?

**Hypothetical RBAC implementation cost:**
- `user_profiles` table with role column (admin / pm / viewer)
- Role management: Slack slash command `/assign-role @user role` (admin-only)
- `PermissionGate` middleware in `src/integrations/slack/event-handler.ts` — analogous to `ApprovalGate`
- Lookup: Slack userId → role via UserStore
- Tests: role-gated paths for all case types

**Estimated complexity:** M-L. The architectural pattern is clear (PermissionGate analogous to ApprovalGate), but implementation requires:
- Slack admin API integration for user group management
- Migration to add `role` column to `user_profiles`
- Role seeding/bootstrapping mechanism
- All existing integration tests must verify they still pass with an admin role seeded

**Business value if built now:** Near-zero. There is no user blocked by the absence of RBAC. Building it preemptively for a future multi-workspace scenario that has not been scheduled is premature optimization.

---

## Design Pre-work (No-Cost Phase 4 Deliverable)

While RBAC is not implemented in Phase 4, the panel recommends producing a design specification document during Sprint J (polish sprint) so that Phase 5 RBAC implementation can begin without design lag. This document should define:

1. Role taxonomy: admin / pm / viewer (minimal 3-tier)
2. PermissionGate interface: `canSubmit(userId: string, caseType: CaseType): Promise<boolean>`
3. UserStore role field: migration DDL
4. Enforcement point: `event-handler.ts` before `IntakeClassifier`
5. Slash command: `/assign-role` (admin only)
6. Test matrix: 3 roles × 7 case types = 21 permission scenarios

Cost: 0.5 sprint (document only, no code). This is a Could-Have for Sprint J.

## Verdict

**APPROVED: RBAC remains Phase 5 (or later). No trigger condition from D-014 has been met.**

**Phase 5 Trigger Conditions (unchanged from D-014, reaffirmed):**
1. Second Slack workspace connected (first external tenant onboarded)
2. President directive to restrict case submission by role for any case type
3. External audit or compliance mandate requiring documented access controls

**Phase 4 Action:** Produce RBAC design specification during Sprint J as a Could-Have (zero-code, zero-risk pre-work for Phase 5).
