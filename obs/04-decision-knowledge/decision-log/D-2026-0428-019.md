## Context

Phase 3 delivered the corpus ingestion pipeline (`src/rag/corpus-ingester.ts`) with the `rag:ingest` npm script for manual CLI execution. The Phase 3 acceptance report explicitly flagged this as a production gap:

> "corpus 重新嵌入触发机制（文件 mtime 跟踪）未实现，KB 文件变更后需手动 rag:ingest。"
> — Phase 3 acceptance report, RAG pipeline correctness dimension, -2 deduction

Phase 4 must close this gap. The question is which ingestion trigger mechanism is appropriate for a production deployment.

---

## Options Evaluated

### Option A — Manual CLI only (current state — no change)
Developer runs `npm run rag:ingest` whenever knowledge base files change.

**Pros:** Zero new code. Simple.
**Cons:** Knowledge base drift. If `agents/pm-agent/expert-knowledge-base.md` is updated, the vector index remains stale until someone manually triggers re-ingestion. In a team setting this is a reliability hazard.

**Verdict:** Rejected. This is a known production gap, not an acceptable long-term state.

### Option B — File watcher (`chokidar`) — recommended
`chokidar` package watches `agents/*/` for changes to `.md` and `.json` files. On detected file change (add/change/unlink): triggers incremental re-ingestion for that file only (idempotent upsert by `agent_id + source_file + chunk_index`). Runs as a background process alongside the Slack bot in development, or as a separate process in production.

**Pros:**
- Zero-friction: developers editing playbooks see changes reflected in RAG without manual steps
- Incremental: only changed file re-embedded, not entire corpus
- Uses existing corpus-ingester infrastructure (add/change = upsert, unlink = delete by source_file)
- `chokidar` is a proven, widely-used file-watching library with minimal overhead
- Can be gated by env var (`ENABLE_CORPUS_WATCHER=true`) for CI safety

**Cons:**
- Only works in environments where the codebase is local to the running process (development / single-server production)
- Does not handle changes made outside the running process (e.g., S3 uploads, git pulls without restart)
- Chokidar is an additional dependency

**Implementation notes:**
- New file: `src/rag/corpus-watcher.ts`
- Entry point: called from `src/index.ts` when `ENABLE_CORPUS_WATCHER=true`
- Watch path: `./agents/` (relative to project root, configurable)
- Debounce: 2000ms to avoid thrashing on rapid saves
- Log to audit: `corpus_watcher.file_changed` event

**Verdict:** Recommended as the primary mechanism.

### Option C — Git pre-commit / post-merge hook
A git hook runs `rag:ingest` on changed agent files before commit or after merge.

**Pros:** Tightly coupled to source control; changes are always ingested before deployment.
**Cons:** Couples embedding to git workflow; requires OpenAI API key in developer environment; significantly slows commits (embedding can take 5-30 seconds per file). Hostile developer experience.

**Verdict:** Rejected. Developer experience cost is too high.

### Option D — S3 / webhook event-driven ingestion
An external event (S3 upload notification, GitHub webhook on push) triggers re-ingestion via API call.

**Pros:** Works in distributed deployments where the file system is remote.
**Cons:** Requires external infrastructure (S3 or webhook receiver) that doesn't exist; over-engineering for current single-server deployment.

**Verdict:** Rejected for Phase 4. Appropriate if deployment moves to container/cloud architecture in Phase 5+.

### Option E — Scheduled re-ingestion (cron)
A cron job runs `rag:ingest` on the full corpus every N hours.

**Pros:** Simple, reliable, no file-watching dependency.
**Cons:** Wasteful (re-embeds unchanged files); delay between file change and index update; embedding cost proportional to corpus size × frequency.

**Verdict:** Rejected as primary mechanism. Acceptable as a fallback / weekly maintenance cron alongside the file watcher.

## Cross-Agent Corpus Retrieval

D-011 phase 3 architectural quality deduction also noted: "cross-agent corpus retrieval not implemented (e.g., ContentAgent querying PMAgent knowledge chunks)."

This is addressed in the same Sprint H work package:

**Current state:** `src/rag/semantic-retriever.ts` scopes queries to `WHERE agent_id = $1`. ContentAgent only sees ContentAgent chunks.

**Phase 4 change:** Add `retrieveFromCorpus(agentId: string, query: string, topK: number)` for single-agent retrieval (existing behavior) AND `retrieveFromAllCorpora(query: string, topK: number, excludeAgentId?: string)` for cross-agent retrieval. The orchestrator can use cross-corpus retrieval for broader context enrichment.

This change requires no schema migration (the `agent_id` column is already present for filtering; removing the filter is trivially a different query).

## Verdict

**APPROVED: Option B (chokidar file watcher) as the primary ingestion trigger for Phase 4.**

### Implementation Contract

| Component | Detail |
|-----------|--------|
| New file | `src/rag/corpus-watcher.ts` |
| Dependencies | `chokidar` (already in package.json as a common dependency; add if absent) |
| Watch path | `./agents/*/` for `**/*.md` and `**/*.json` |
| Trigger behavior | Debounce 2000ms → incremental upsert for changed file only |
| Unlink behavior | Delete chunks by `source_file` from `knowledge_chunks` table |
| Guard | `ENABLE_CORPUS_WATCHER=true` env var required (default: off) |
| `.env.example` update | Add `ENABLE_CORPUS_WATCHER=false  # Set to true in dev to auto-re-ingest on KB file changes` |
| Audit log event | `corpus_watcher.file_changed` with `{ sourceFile, agentId, chunksUpserted }` |
| Cross-corpus retrieval | Add `retrieveFromAllCorpora()` method to `src/rag/semantic-retriever.ts` |
| Fallback | `rag:ingest` CLI script remains for full re-ingestion and initial setup |

### Phase 5 Upgrade Path
If deployment moves to containerized/cloud infrastructure, replace file watcher with an S3 event notification or git push webhook triggering the same `corpus-ingester.ts` logic via HTTP endpoint. The ingester interface remains unchanged.
