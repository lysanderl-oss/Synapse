## Context

Phase 1 delivered 3 core scenarios (weekly_report / meeting_prep / internal_service), 83 tests, 0 tsc errors, and a clean governance chain. The Phase 1 acceptance report listed 5 Phase 2 candidates without prioritization. The codebase was inspected directly. The original requirements doc (Synapse business scenarios) defines the north star as "a governable digital twin collaboration system."

The Phase 2 candidates under consideration are:
1. PostgreSQL + pgvector migration (from file-based FileStore)
2. 4 additional scenarios: research_request / content_draft / data_analysis / decision_brief
3. Real Slack Bolt integration verification (beyond FakeBoltApp)
4. EvidenceBundle auto-persist on approval event
5. Cross-channel delivery routing
6. Role-based access control (RBAC)
7. Web dashboard / reporting layer

## Options Evaluated

**Option A â€” Scenarios-first:** Add all 4 scenarios in Phase 2, defer database migration.
**Option B â€” Infrastructure-first:** Do PostgreSQL migration first, then scenarios.
**Option C â€” Balanced (recommended):** EvidenceBundle fix + 4 scenarios + Bolt smoke test as Must-Have; PostgreSQL migration as Should (Sprint C); RBAC and dashboard deferred to Phase 3.

## Rationale

- The 4 new scenario types (research_request / content_draft / data_analysis / decision_brief) are **already fully wired** in `CaseType` enum, `case-types.yaml`, `routing-rules.yaml`, `sla-config.yaml`, and `IntakeClassifier`. The routing and FSM infrastructure is 100% ready. Adding these scenarios is playbook + agent handler work only â€” very low risk.
- The EvidenceBundle handler gap (approval action does not auto-persist EvidenceBundle) is a **data integrity defect** documented in WP4 honest caveats. It must be corrected before Phase 2 adds more scenarios.
- PostgreSQL migration is **not blocking** any Phase 2 scenario. File-based storage is functionally complete. The migration adds operational resilience for production scale. Appropriate for Sprint C (after scenarios proven on the new agent pattern).
- Slack real Bolt integration: the current FakeBoltApp faithfully tests the listener registration contract. A socket-mode smoke test (or nock-based shim) is needed for CI confidence but is S-size work, appropriate as Phase 2 Sprint B gate.
- RBAC is not needed for a single-tenant deployment. The current `authorized_tools` field in `AgentRole` already provides agent-level permission boundary. Human RBAC adds complexity without current business need. Phase 3.
- Web dashboard: no user story driving it. Phase 3.

---

## Verdict

**Phase 2 Must-Have (Sprint B + C):**
- WP0: EvidenceBundle auto-persist fix + ServiceAgent multi-case-type expansion scaffold
- Sprint B: 4 new scenarios (research_request / content_draft / data_analysis / decision_brief) via ResearchAgent + ContentAgent pattern
- Sprint B: Slack Bolt smoke test (socket mode contract, CI gate)
- Sprint C: PostgreSQL migration (adapter pattern, down migration required)

**Phase 3 (deferred):**
- RBAC / multi-tenant authorization
- Web dashboard / reporting layer
- pgvector / RAG (after PostgreSQL migration is stable)
- Cross-org multi-agent handoff
