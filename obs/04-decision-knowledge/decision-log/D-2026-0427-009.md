# D-2026-0427-009 — Phase 2 New Agent Architecture (ResearchAgent + ContentAgent Pattern)

- **Date:** 2026-04-27
- **Decision Level:** L3 (Lysander + Expert Panel)
- **Status:** APPROVED
- **Decided by:** Expert Panel (product_strategist + harness_engineer + integration_qa)
- **Related:** D-2026-0427-007 (scope), D-2026-0427-008 (storage)

---

## Context

Phase 2 adds 4 new scenarios: research_request / content_draft / data_analysis / decision_brief. All 4 are already registered in `CaseType`, `case-types.yaml`, and `routing-rules.yaml` with `owner_agent: agent.ops`. The question is whether each scenario gets a dedicated agent class or whether a generalized agent with scenario-specific playbooks handles multiple types.

Phase 1 established two patterns:
- **PMAgent** (`agent.pm_zh`): handles `weekly_report` and `meeting_prep` — two thematically related scenarios with shared project-management framing
- **ServiceAgent** (`agent.ops`): handles `internal_service` only

The Phase 1 `ServiceAgent` was intentionally kept as a lightweight peer of PMAgent (WP4 honest caveats). Its `run()` method is a single-type guard (`if caseEntity.type !== CaseType.INTERNAL_SERVICE return error`). This pattern does not scale cleanly to 4 new types under `agent.ops`.

---

## Options Evaluated

**Option A — One agent per scenario (4 new classes):** ResearchRequestAgent, ContentDraftAgent, DataAnalysisAgent, DecisionBriefAgent.
- Pro: Maximum isolation, easy to test, easy to evolve independently.
- Con: High boilerplate for nearly identical code structure; the 4 agents share 80%+ of their implementation (safeRead helpers, callClaude wiring, EvidenceBundle persistence).
- Verdict: Rejected unless scenarios diverge significantly in execution pattern.

**Option B — Two new generalized agents (recommended):** ResearchAgent (handles `research_request` + `data_analysis`) and ContentAgent (handles `content_draft` + `decision_brief`). Each uses scenario-specific playbooks and templates loaded by `CaseType` dispatch inside `run()`.
- Pro: Follows the PMAgent precedent (one agent, multiple related types). Reduces boilerplate. Each agent still has clear thematic coherence: ResearchAgent = analytical/investigative work; ContentAgent = writing/synthesis work.
- Con: Slightly more complex `run()` dispatch logic inside each agent — but this is already present in PMAgent.
- Verdict: Recommended.

**Option C — Extend ServiceAgent to handle all 4 new types:** Make `agent.ops` a catch-all ops agent.
- Con: Violates single-responsibility principle. ServiceAgent would balloon. Testing becomes unwieldy.
- Verdict: Rejected.

---

## Verdict

**APPROVED: Option B — ResearchAgent + ContentAgent with playbook dispatch.**

Implementation:
- `src/agents/research-agent.ts` — `agent.research` (agentId), handles `research_request` + `data_analysis`
- `src/agents/content-agent.ts` — `agent.content` (agentId), handles `content_draft` + `decision_brief`
- `agents/research-agent/playbooks/research-request.md`
- `agents/research-agent/playbooks/data-analysis.md`
- `agents/research-agent/templates/research-request.md`
- `agents/research-agent/templates/data-analysis.md`
- `agents/content-agent/playbooks/content-draft.md`
- `agents/content-agent/playbooks/decision-brief.md`
- `agents/content-agent/templates/content-draft.md`
- `agents/content-agent/templates/decision-brief.md`

**config/case-types.yaml update required:**
- `research_request.owner_agent: agent.research`
- `data_analysis.owner_agent: agent.research`
- `content_draft.owner_agent: agent.content`
- `decision_brief.owner_agent: agent.content`

**config/routing-rules.yaml update required:** same agent_id updates.

**AgentRegistry update:** register ResearchAgent and ContentAgent instances at startup.

**Rationale for separate agent IDs vs. reusing `agent.ops`:** The Phase 1 `agent.ops` is the ServiceAgent's identity. Reusing it for research and content work would confuse routing and UserDossier attribution. New agent IDs are cleaner and allow independent capability evolution.
