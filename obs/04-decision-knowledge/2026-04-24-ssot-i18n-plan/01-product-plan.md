# 方案 ①：产品团队方案 — SSOT 与双语作为定位题

## BLUF

**SSOT 本质不是技术问题，是品牌可信度问题。**
**双语本质不是翻译问题，是两个市场两条 GTM 的定位题。**

若把 SSOT 当作"把 synapse-core 内容复制到网站一份"的工程问题处理，我们只解决了一次同步，没解决"Synapse 的公开技术文档究竟是什么"的定位问题。若把双语当作"把中文文章翻译成英文"的内容问题处理，我们会在错误的地方堆资源 —— 中文市场与英文市场的用户画像、购买决策链、关键词生态完全不重叠。

本方案从产品定位角度给出建议：SSOT 应建立在"Synapse 公开 API 边界"之上（契约先于实现）；双语应建立在"两个 GTM 的共同骨架 + 差异化纵深"之上（骨架双语，纵深按市场）。

## 一、现状盘点

### 1.1 学院现状（synapse.janusd.com/learn）

- 总页数：7 页
- 语言：100% 中文
- 内部链接错误：约 30% 指向已废弃的 `/ai-team-system/*` 路径（该栏目在 Synapse v2.5 重组后已下线，但学院内链未更新）
- 定位模糊：既像使用手册，又像新手引导，还像完整课程，三者都不饱和

### 1.2 Forge 现状（synapse-forge.com/get-started）

- 与学院同病：中文为主，链接指向已废弃页面
- get-started 5min 与 30min 两个版本未区分，同一页面混合两种节奏
- 与 synapse-core 仓库实际目录结构存在漂移（示例命令中的文件路径已过时）

### 1.3 博客现状

- 总文章数：33 篇
- 语言：100% 中文
- 平均字数：约 2400 字
- 作者标签混乱：部分使用"Lysander"、部分使用"Multi-Agents"、部分未标注
- 标签体系缺失：无统一 tags 字典，无法按主题聚合

### 1.4 内容源真相

synapse-core 仓库是所有 Synapse 配置、Skill、执行链、HR 模板的唯一正确源。当前三处对外呈现（学院、Forge、博客）对 synapse-core 的引用方式不一致：

- 学院：手抄版本（脱钩严重）
- Forge：半引用半手抄（部分示例直接嵌入，部分复述）
- 博客：零引用（全部是作者视角叙事）

## 二、SSOT 建议

### 2.1 推荐方案：B — Git Submodule

**注：此为产品方案原始推荐，与技术方案 C 存在分歧，已由智囊团裁决统一采用技术方案 C（Build-time Extract 脚本）。保留此节作为决策溯源。**

产品团队原建议理由：
- Submodule 锁定 synapse-core commit，保证任意时刻学院/Forge 展示的内容都可溯源
- 开发者熟悉 submodule 工作流，培训成本低
- 与 GitHub 生态天然集成，issue/PR 可跨仓库引用

产品团队未采用的方案：
- A（直接复制）：最高维护成本，最易脱钩
- C（Extract 脚本）：引入独立构建步骤，产品团队认为会延长发布周期
- D（npm package）：synapse-core 非代码，打 npm 包语义不匹配
- E（Headless CMS）：增加第三方系统依赖，与 Synapse 自主可控原则冲突

### 2.2 SSOT 公开边界

无论最终技术路径是 B 还是 C，产品团队强调：**synapse-core 仓库必须显式划出公开 API 边界**。建议目录 `docs/public/`，该目录下文件视同对外发布，PR merge 前需走"公开内容变更 Review"流程。非 `docs/public/` 目录下的文件视为内部配置，不得被学院/Forge 引用。

## 三、学院定位

### 3.1 推荐方案：使用手册 + 社区

**注：此为产品方案原始推荐，与内容团队方案存在分歧，已由智囊团裁决采用内容方案"课程+认证+下载中心"。保留此节作为决策溯源。**

产品团队原建议：
- 学院作为 Synapse 使用手册的门户（索引 + 搜索 + 按场景的任务指南）
- 社区（forum 或 Discord embed）作为使用者互助渠道
- 不强推认证体系（认证需要法律、合规、考试系统三套基础设施，学院当前不具备）

产品团队的担忧：课程+认证定位会把学院绑定到教育产品赛道，后续若 Synapse 商业化重心转向企业 Agent 治理平台，学院的教育属性会成为包袱。

## 四、URL i18n 策略

### 4.1 推荐方案：栏目差异化默认语言

**注：此为产品方案原始推荐，与技术方案存在分歧，已裁决采用技术方案"全站统一 / 默认中文 + /en/ 英文前缀"。保留此节作为决策溯源。**

产品团队原建议：
- Forge（面向全球 Builder 与企业决策者）：英文默认，`/zh/` 中文前缀
- 主站 synapse.janusd.com（面向中文学习者与团队伙伴）：中文默认，`/en/` 英文前缀
- 理由：尊重两个栏目的实际受众画像，避免"一刀切默认中文"对英文 Builder 造成心理距离

产品团队未采用方案：全站统一中文默认（技术方案建议）—— 产品团队认为这会在 Forge 上丢失英文 Builder 信任。

## 五、双语范围

### 5.1 优先级矩阵

- **P0（首期双语）**：Forge 全站骨架、学院核心 5 页、商业页（pricing / about / contact / privacy / terms）
- **P1（二期双语）**：学院深度页、博客 Top 8（按过去 12 个月流量排名）
- **P2（长尾）**：博客剩余 25 篇、SOP、内部方法论

### 5.2 博客双语原则

- 不做存量机翻
- 仅对高价值内容（P1）做再创作级双语
- 长尾保留中文，英文读者通过 canonical link 跳转

## 六、用户影响矩阵

| 用户群 | SSOT 改造影响 | 双语改造影响 |
|--------|--------------|--------------|
| 团队伙伴（中文） | 学院链接修复后可用 | 无影响（仍默认中文） |
| Builder（英文） | Forge 示例可执行 | Forge 英文后可进入 |
| 企业决策者 | pricing/about 可信度提升 | 商业页英文后可评估 |
| 中文学习者 | 学院手册可用 | 无影响 |
| 搜索引擎（SEO） | 内链结构修复提升抓取 | hreflang 正确可翻倍索引 |

## 七、成功标准

1. 学院/Forge 内链错误率 = 0（当前约 30%）
2. synapse-core 内容在学院/Forge 的漂移延迟 < 24h
3. Forge 英文 PV 占比 ≥ 40%
4. 学院中文 PV 环比提升 ≥ 20%
5. 博客 Top 8 英文版本上线后，英文自然搜索 CTR > 2%
6. 总裁不需要回答"学院那个链接为什么 404"类问题

## 八、未决问题（需总裁决策）

1. synapse-core 是否立即公开到 GitHub？（影响 SSOT 工程路径选择）
2. SCP（Synapse Certified Professional）课程暂时占位还是正式开课？
3. 栏目语言默认不同（Forge 英文默认 / 主站中文默认）是否可接受？
4. 博客作者署名风格是否统一为"Multi-Agents"？
